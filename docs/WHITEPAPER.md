> 以Case为中心的AI-Native安全运营Agent框架

## 1. 行业背景与挑战

### 1.1 安全运营的困境

随着企业数字化转型的深入，网络安全威胁的规模和复杂性持续攀升。安全运营团队面临前所未有的挑战：

**告警海啸**：中大型企业SOC日均处理告警量从数千到数万条不等，其中超过90%为误报或低价值告警。分析师被迫在"告警海洋"中寻找真正的威胁，效率极低。

**人才缺口**：全球网络安全人才缺口已超过400万。一名合格的SOC分析师需要2-3年的培养周期，而安全团队的人员流动率居高不下。企业面临"招不到、留不住"的困境。

**工具碎片化**：典型SOC使用10-20种安全工具（SIEM、EDR、NDR、WAF、TIP、漏洞扫描器等），工具间缺乏有效联动。分析师需要在多个平台间频繁切换，手动复制粘贴数据，上下文在切换中丢失。

**响应时间窗口不对称**：攻击者平均在数分钟内完成横向移动，而企业从告警产生到完成处置的平均时间（MTTR）通常以天甚至周计。这种时间窗口的严重不对称，使得即便检测能力到位，响应也往往滞后。

### 1.2 AI带来的范式变革

大语言模型（LLM）和AI Agent技术的快速成熟，为安全运营带来了范式变革的可能：

- LLM具备理解安全上下文、分析攻击手法、生成处置建议的能力
- AI Agent能够自主规划任务、调用工具、执行多步骤工作流
- MCP等标准协议使得AI与安全工具的集成变得标准化和可扩展

**安全运营正从"人工驱动"向"AI驱动+人工监督"演进。**

## 2. 产品理念：AI-Native安全运营

### 2.1 什么是AiCSO

AiCSO（AI Cyber Security Operations，AI安全运营）是一个以Case为中心的AI-Native安全运营Agent框架。它不是在传统SOC工具上叠加一个"AI助手"，而是**用AI重构安全运营的每一个环节**。

### 2.2 核心理念

**Case为中心**：所有安全运营工作围绕Case展开。Case是事件调查、响应处置、协作沟通、知识沉淀的核心载体。打破了传统工具以"告警"或"日志"为中心的局限，提供了统一的安全事件视图。

**AI-Native**：AI不是辅助工具，而是核心驱动力。从告警接入的那一刻起，AI Agent就开始工作——聚合告警、研判真伪、关联上下文、推荐响应方案。人类分析师从"执行者"转变为"决策者"和"监督者"。

**Agent协作**：模拟真实SOC团队的协作模式。多个专业Agent各司其职——分诊Agent负责告警分类，调查Agent负责深入分析，响应Agent负责执行处置——由编排Agent统一调度，协同完成复杂的安全运营任务。

**开源开放**：完全开源，Apache 2.0协议。避免厂商锁定，支持社区共建。安全运营的最佳实践不应该被少数厂商垄断。

### 2.3 设计原则

| 原则 | 说明 |
|------|------|
| **人在回路（Human-in-the-loop）** | AI做分析和建议，人做最终决策。高风险操作必须人工审批 |
| **可解释性** | Agent的每一个决策都有推理过程，分析师可以理解AI为什么这么判断 |
| **渐进式自动化** | 从辅助分析开始，逐步扩展自动化范围。用户始终控制自动化程度 |
| **插件化架构** | 数据源、工具、Agent、Playbook均为可插拔组件，支持社区扩展 |
| **安全优先** | 框架本身的安全性是第一要务。防Prompt注入、最小权限、完整审计 |

## 3. 核心能力

### 3.1 Case生命周期管理

AiCSO以Case为核心，提供完整的安全事件管理能力：

```
告警流入 → 自动聚合 → 智能分诊 → 深入调查 → 响应处置 → 复盘关闭
   │          │          │          │          │          │
   ▼          ▼          ▼          ▼          ▼          ▼
 数据源     聚合引擎    Triage     Investigation Response   知识沉淀
 适配器    (规则+AI)    Agent       Agent       Agent      (案例库)
```

**关键特性**：
- **智能聚合**：多个相关告警自动聚合为一个Case，避免告警碎片化
- **AI分诊**：Agent自动研判告警真伪，给出置信度评分和初步分析
- **统一视图**：一个Case汇聚所有相关信息——告警、资产、IoC、时间线、处置记录
- **状态管理**：完整的Case状态机，支持工作流驱动的事件管理
- **SLA管理**：基于严重级别的响应时间要求，超时自动升级

### 3.2 告警接入与智能研判

**统一接入**：通过数据源适配器，支持接入任意安全设备的告警。内置Syslog、CEF、LEEF等通用解析器，以及SIEM、EDR等平台的API适配器。

**双引擎聚合**：
- **规则引擎**：基于IP、资产、时间窗口等维度的确定性聚合，快速、可预测
- **AI引擎**：基于语义理解的智能聚合，能识别规则引擎难以覆盖的关联关系

**智能研判**：
- Agent分析告警上下文，判断是否为真阳性
- 自动关联资产信息、威胁情报、历史案例
- 生成人类可读的研判报告，包含推理过程
- 给出置信度评分和推荐处置方案

### 3.3 响应编排与分级审批

**Playbook驱动**：标准化的响应流程通过Playbook定义，支持YAML声明式配置。内置常见场景的Playbook模板（钓鱼邮件、暴力破解、恶意软件等）。

**分级审批**：
- **低风险**（查询情报、检索信息）：Agent自动执行
- **中风险**（封禁IP、召回邮件）：需要L3分析师审批
- **高风险**（隔离主机、修改防火墙规则）：需要SOC管理者审批

**完整审计**：所有Agent操作和人工操作均有完整审计日志，满足合规要求。

### 3.4 可扩展的工具生态

**MCP协议支持**：通过Model Context Protocol标准协议接入任意工具和数据源，无需修改框架代码。

**Skill插件体系**：将相关的工具、Playbook、知识封装为Skill，支持社区共享和复用。

**自定义Agent**：基于框架提供的Agent基类，用户可以开发满足特定需求的自定义Agent。

## 4. 技术优势与差异化

### 4.1 与传统SOAR的差异

| 维度 | 传统SOAR | AiCSO |
|------|---------|-------|
| **核心范式** | 规则驱动的自动化 | AI驱动的智能运营 |
| **告警处理** | 基于预定义规则的匹配和路由 | AI理解上下文，智能研判和聚合 |
| **响应能力** | 执行预定义Playbook | Agent自主规划响应方案，结合Playbook执行 |
| **适应性** | 需要人工编写和维护规则 | Agent能处理未见过的攻击场景 |
| **学习能力** | 无 | 从历史Case中检索参考（已实现），自适应优化研判模型（规划中） |
| **部署门槛** | 重（需要完整基础设施） | 轻（CLI即可开始使用） |
| **成本** | 高（商业许可费用） | 零（开源） |

### 4.2 与国内安全产品的差异

国内主流安全厂商（奇安信、深信服、绿盟、安恒等）的态势感知/SOC平台通常采用传统架构，与AiCSO的差异如下：

| 维度 | 国内态势感知平台 | AiCSO |
|------|----------------|-------|
| **AI能力** | 基于规则和统计模型，AI作为辅助 | AI-Native，Agent驱动整个运营流程 |
| **架构** | 单体/微服务，预定义流程 | Agent框架，灵活编排 |
| **告警处理** | 规则匹配+关联分析 | 规则+AI双引擎，语义级理解 |
| **响应能力** | 固定Playbook，人工触发 | Agent自主规划，分级审批自动执行 |
| **部署模式** | 重资产部署，通常需要专业服务 | 轻量起步（CLI），按需扩展 |
| **开放性** | 封闭生态，厂商锁定 | 开源开放，MCP标准协议 |
| **定制成本** | 高（需要厂商支持） | 低（社区驱动，插件化） |
| **适用场景** | 大型企业，需要完整SOC平台 | 从中小企业到大型SOC团队均可 |

AiCSO并非要替代这些平台，而是提供一种更灵活、更智能的补充方案。通过MCP协议和API适配器，AiCSO可以对接现有态势感知平台的数据源，在其之上构建AI驱动的智能运营层。

### 4.2 Agent架构优势

**专业化分工**：每个Agent专注于特定领域（分诊、调查、响应等），而非试图用一个Agent解决所有问题。这模拟了真实SOC团队的分工协作模式。

**可组合性**：Agent可以灵活组合，适应不同的安全运营场景。新增能力只需添加新Agent，不影响已有系统。

**可解释性**：每个Agent的推理过程透明可见，分析师可以理解AI的判断依据，而非面对一个"黑盒"。

**可控性**：通过分级审批机制，用户始终控制Agent的行为边界。从"完全人工"到"完全自动"之间，有多个可选的自动化级别。

### 4.3 开源优势

- **零成本起步**：无许可证费用，中小企业也能使用先进的安全运营能力
- **透明可信**：代码完全开放，安全团队可以审计框架本身的安全性
- **社区驱动**：安全从业者共同贡献检测规则、Playbook模板、数据源适配器
- **无厂商锁定**：数据和配置完全自主控制，不依赖任何任何特定厂商
- **灵活定制**：基于开源框架进行二次开发，满足企业特定需求

### 4.4 风险与应对

我们清醒地认识到AI在安全运营中的局限性，并在架构设计中充分考虑了这些风险：

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| **LLM幻觉** | AI可能生成错误的研判结论 | 置信度评分机制，低置信度结果强制人工审核；所有AI决策可追溯推理过程 |
| **LLM服务中断** | 依赖外部API，可能不可用 | 多Provider Fallback机制；支持本地模型部署（Ollama）；LLM不可用时降级为规则引擎 |
| **Prompt注入** | 恶意日志可能影响AI判断 | 多层防护（模式匹配+指令分离+权限沙箱+输出验证）；Agent权限最小化 |
| **误判成本** | 自动响应可能造成业务影响 | 分级审批机制，高风险操作必须人工确认；所有响应操作可回滚 |
| **Token成本** | LLM调用费用可能较高 | 模型分级策略（简单任务用轻量模型）；缓存机制；支持本地开源模型 |
| **数据隐私** | 告警数据发送到外部LLM | 支持本地模型部署；敏感字段自动脱敏；数据不出域 |

AiCSO的设计原则是：**AI负责分析和建议，人负责决策和监督**。系统始终保留人工兜底能力。

## 5. 典型应用场景

### 5.1 场景一：钓鱼邮件事件

**传统流程**（约2-4小时）：
1. 分析师收到邮件网关告警
2. 手动查看邮件内容和附件
3. 手动提取URL和附件Hash
4. 在多个情报平台逐一查询
5. 判断是否为钓鱼邮件
6. 如确认，手动召回邮件、封禁发件人
7. 检查是否有员工已点击
8. 撰写事件报告

**AiCSO流程**（AI处理时间15-30分钟，不含人工审批等待）：
1. 告警自动接入，创建Case
2. TriageAgent自动研判：分析邮件头、URL、附件 → 判断为钓鱼邮件（置信度95%）
3. InvestigationAgent自动调查：查询情报、检索历史、检查邮件网关日志
4. ResponseAgent推荐响应方案：召回邮件 + 封禁发件人 + 检查点击记录
5. 分析师审核AI建议，一键批准执行（中风险操作需L3审批）
6. ReportAgent自动生成事件报告

### 5.2 场景二：暴力破解攻击

**传统流程**：分析师手动关联多条登录失败告警 → 判断攻击模式 → 手动封禁IP → 检查账号状态

**AiCSO流程**：
1. 聚合引擎自动将同一源IP的多条登录失败告警聚合为一个Case
2. Agent判断为暴力破解攻击，关联威胁情报确认IP为恶意
3. 推荐封禁IP + 锁定目标账号
4. 分析师确认后自动执行

### 5.3 场景三：内部威胁狩猎

**传统流程**：L3分析师凭经验在SIEM中编写查询 → 手动分析结果 → 多轮迭代

**AiCSO流程**：
1. 分析师描述狩猎假设："查找可能的数据外传行为"
2. InvestigationAgent自主规划调查步骤：查询异常外联、大流量传输、敏感数据访问
3. Agent执行多轮调查，逐步缩小范围
4. 生成调查报告，标注可疑行为和证据链

## 6. 部署与集成

### 6.1 部署模式

**轻量部署（开发/POC）**：
```bash
pip install aicso
aicso config set llm.provider openai
aicso config set llm.api_key sk-xxx
aicso datasource add --type syslog --port 514
# 即可开始使用
```

**标准部署（生产）**：
```bash
git clone https://github.com/aicso/aicso.git
cd aicso
docker compose up -d
# 包含：AiCSO + PostgreSQL + Redis + Qdrant + Elasticsearch
```
